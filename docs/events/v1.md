# Event and Command Model (v1)

## Purpose
Pagooz separates domain events, delivery commands, and ledger commands to avoid semantic mixing.

## Message Categories
### 1. Domain events
Canonical business events emitted by Core and Ledger components.
Examples:
- `payment_intent.created`
- `quote.created`
- `checkout_session.created`
- `ledger.journal_posted`

Transport queue:
- `q-domain-events-{env}` for Core domain events
- `q-ledger-events-{env}` for Ledger lifecycle events

### 2. Ledger posting commands
Accounting commands emitted by Core for Ledger processing.
Example:
- `ledger.post_entries`

Transport queue:
- `q-ledger-commands-{env}`

### 3. Delivery commands
Notification and webhook delivery workloads.

Transport queues:
- `q-notification-outbox-{env}`
- `q-webhook-outbox-{env}`

Delivery queues are not used as generic domain event buses.

## Envelope Principles
Event envelopes include:
- `event_id`
- `event_type`
- `occurred_at`
- `tenant_id`
- `mode`
- resource references
- event payload data
- visibility metadata

## Ledger Event Semantics
`ledger.journal_posted` indicates a balanced journal was persisted and balances updated.
`ledger.journal_rejected` indicates a command failed validation and was acknowledged as invalid.

## Idempotency and Ordering
- Ledger command idempotency is enforced by `journal_id`.
- Consumers must tolerate retries and duplicate deliveries.
- Producers should generate deterministic references for replay safety.
